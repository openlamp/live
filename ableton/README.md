# Ableton assets

Mode A assets. Everything derives from **`mapping.spec.json`** ‚Äî the machine-readable
mirror of the [LumiDeck MIDI convention](https://github.com/openlamp/midi/blob/main/MIDI-PROTOCOL.md)
(source of truth; do not diverge). See [../docs/DESIGN.md](../docs/DESIGN.md).

| Path | What | Status |
|---|---|---|
| `mapping.spec.json` | channels‚Üígroups, notes‚Üíactions, CC‚Üíparams | ‚úÖ |
| `clips/*.mid` | draggable MIDI clips, generated by [`../tools/gen_clips.py`](../tools/gen_clips.py) | ‚úÖ 19 clips |
| `OpenLamp-demo.als` | demo set: one MIDI track per group, pre-routed to `LumiDeck` on its channel | ‚òê author once in Live |

## The clips

Regenerate anytime:

```
python3 tools/gen_clips.py --verify
```

- `note*.mid` ‚Äî one note-on each = fire a colour / power state / animation. These
  import reliably into Live.
- `cc*-sweep.mid` ‚Äî a CC ramp (brightness, hue) documenting the intended envelope.
  Live's `.mid` import may not carry CC as automation; redraw as clip automation if
  so (the ramp shape is the reference).

Drop a clip onto a MIDI track routed to `LumiDeck` (see
[../docs/ABLETON-SETUP.md](../docs/ABLETON-SETUP.md)). The **track's channel** picks
the lamp group, so clips are written on channel 1 and inherit the routing.

## Why no `.als`/`.adg` in git yet

Those are Ableton's undocumented, version-fragile binaries ‚Äî synthesizing them
headless would be guesswork. The `.als` template is exported once from Live; the
`.mid` clips (open format) carry the reproducible, generated part.
