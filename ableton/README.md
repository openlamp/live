# Ableton assets

Mode A assets. Everything derives from **`mapping.spec.json`** — a pinned mirror of
the [wled-midi](https://github.com/openlamp/wled-midi) core (note/CC numbers). The
wire semantics + value transforms are the spec's; do not diverge. See
[../docs/DESIGN.md](../docs/DESIGN.md).

| Path | What | Status |
|---|---|---|
| `mapping.spec.json` | pinned note/CC numbers (labels for clip names) | ✅ wled-midi v0.3.0 |
| `clips/*.mid` | draggable MIDI clips, generated by [`../tools/gen_clips.py`](../tools/gen_clips.py) | ✅ 19 clips (looks/util/modifiers) |
| `OpenLamp-demo.als` | demo set: one MIDI track per target, pre-routed to the lamp port on its channel | ☐ author once in Live |

## The clips

Regenerate anytime:

```
python3 tools/gen_clips.py --verify
```

- `note*.mid` — one note-on each = fire a colour / power state. These import
  reliably into Live.
- `cc*-sweep.mid` — a CC ramp (brightness, hue) documenting the intended envelope.
  Live's `.mid` import may not carry CC as automation; redraw as clip automation if
  so (the ramp shape is the reference).

Drop a clip onto a MIDI track routed to the lamp port (see
[../docs/ABLETON-SETUP.md](../docs/ABLETON-SETUP.md)). The **track's channel** picks
the target, so clips are written on channel 1 and inherit the routing.

## Why no `.als`/`.adg` in git yet

Those are Ableton's undocumented, version-fragile binaries — synthesizing them
headless would be guesswork. The `.als` template is exported once from Live; the
`.mid` clips (open format) carry the reproducible, generated part.
